# Go live — full deployment checklist

This app is **two services**: a React frontend and a FastAPI API with a database and file uploads.

**Important:** Vercel alone cannot run this backend as-is. Vercel is for static sites and short-lived serverless functions. This API uses **SQLite on disk** and **local file uploads** — those need a always-on server (Render, Railway, Fly.io, etc.).

The setup below gives you **one public URL on Vercel** for users, with the API hosted on Render (free tier works for demos). Everything (login, tasks, attachments, reports) works.

---

## What you will have when done

| Piece | Where it runs | URL |
|-------|----------------|-----|
| React app | Vercel | `https://your-app.vercel.app` |
| FastAPI API | Render | `https://your-api.onrender.com` |
| Database | Render (SQLite file) | (internal) |
| Uploads | Render disk | served at `/uploads` on API |

## Part 1 — Deploy the API (Render) ~15 min

### 1. Push code to GitHub

```bash
git add .
git commit -m "Prepare for deployment"
git push origin main
```

### 2. Create the API on Render

1. Open [dashboard.render.com](https://dashboard.render.com)
2. **New** → **Blueprint**
3. Connect your GitHub repo (`notion-task-tracker`)
4. Render reads `render.yaml` and creates a web service `notion-task-tracker-api`
5. Click **Apply**

### 3. Set environment variables (Render → your service → Environment)

Set these **before** or right after the first deploy:

| Key | Value | Notes |
|-----|--------|--------|
| `SECRET_KEY` | (auto-generated by Render) | Keep secret |
| `DEBUG` | `false` | |
| `DATABASE_URL` | `sqlite:///./workmanager.db` | Default in render.yaml |
| `AUTO_SEED` | `true` | Creates demo users/tasks on first start |
| `FRONTEND_URL` | `https://YOUR-APP.vercel.app` | Set after Part 2, then redeploy |
| `CORS_ORIGINS` | `https://YOUR-APP.vercel.app` | Same as above |

Use your **exact** Vercel URL (no trailing slash). Preview deploys: add them comma-separated, e.g.  
`https://my-app.vercel.app,https://my-app-git-main-you.vercel.app`

### 4. Wait for deploy and test

Copy the Render URL (e.g. `https://notion-task-tracker-api.onrender.com`).

Open in browser:

```
https://YOUR-API-URL.onrender.com/api/health
```

Expected: `{"success":true,"data":{"status":"healthy"},...}`

**First request after idle** on free tier may take 30–60 seconds (cold start).

Demo login (after auto-seed):

| Role | Email | Password |
|------|-------|----------|
| Admin | admin@company.com | admin123 |

## Part 2 — Deploy the frontend (Vercel) ~10 min

### 1. Import project

1. [vercel.com/new](https://vercel.com/new)
2. Import the **same GitHub repo**
3. Configure:

| Setting | Value |
|---------|--------|
| **Root Directory** | `frontend` ← required |
| **Framework Preset** | Vite |
| **Build Command** | `npm run build` |
| **Output Directory** | `dist` |

### 2. Environment variable

Add in Vercel → **Settings** → **Environment Variables**:

| Name | Value | Environments |
|------|--------|----------------|
| `VITE_API_URL` | `https://YOUR-API-URL.onrender.com` | Production, Preview, Development |

**No trailing slash.** Example: `https://notion-task-tracker-api.onrender.com`

### 3. Deploy

Click **Deploy**. When finished, open `https://your-app.vercel.app`.

### 4. Finish CORS on Render

1. Copy your Vercel production URL
2. Render → Environment → set `FRONTEND_URL` and `CORS_ORIGINS` to that URL
3. **Manual Deploy** → Deploy latest commit on Render

## Can everything run *inside* Vercel only?

**Not with the current codebase**, without major changes:

| Current feature | Vercel limitation | What you’d need |
|-----------------|-------------------|-----------------|
| SQLite database | No persistent disk on serverless | PostgreSQL (Neon, Supabase, Vercel Postgres) |
| File uploads to `./uploads` | Ephemeral filesystem | Vercel Blob or S3 |
| Long-running FastAPI | Serverless time limits | Rewrite app for serverless (e.g. Mangum) + Postgres |

That is a **multi-day migration**, not a config change. For a **fully working live app today**, use **Vercel + Render** as above.

For **production** (real team, data must not disappear):

- Move database to **PostgreSQL** (Render Postgres, Neon, Supabase)
- Use **persistent storage** or object storage for uploads
- Render **paid** plan or Railway with a volume if you keep SQLite temporarily

## Troubleshooting

### Login fails / “Network Error”

- Confirm `VITE_API_URL` on Vercel matches Render URL exactly (https, no trailing slash)
- Redeploy Vercel after changing env vars
- Open browser DevTools → Network: API calls should go to `https://....onrender.com/api/...`

### CORS error in console

- Add your Vercel URL to `CORS_ORIGINS` on Render
- Redeploy Render after changing env vars

### API very slow first time

- Render free tier sleeps after ~15 min idle; first hit wakes it up (~30–60 s)

### Data disappeared after redeploy

- Free Render disk is **not guaranteed persistent**; use paid disk or PostgreSQL for production

### 404 when refreshing `/my-tasks` or `/reports`

- Ensure `frontend/vercel.json` is in the repo (SPA rewrite to `index.html`)
- Redeploy Vercel

### Attachments don’t work

- Uploads live on the **API server**; `VITE_API_URL` must point to Render, not only Vercel
- After API redeploy, uploaded files on free tier may be lost (same disk issue)
